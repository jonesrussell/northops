# M2-1: Brand Voice & Messaging Guide — NorthOps

> Governs all NorthOps copywriting: website, outreach, proposals, social, and internal docs.
> Grounded in M1 outputs: positioning, service lines, client segments, and differentiation.

---

## Voice Principles

1. **Direct.** Say what you mean in the fewest words. Lead with the point, not the preamble. If a sentence doesn't add information, cut it.

2. **Technical.** Write for people who build things. Use precise language — "CI/CD pipeline," not "streamlined workflows." Name the stack, the tool, the pattern. Clients hire NorthOps because we know the work, and the writing should reflect that.

3. **Grounded.** Every claim is backed by something real — a skill, a deliverable, a price, a timeline. No aspirational language, no future promises. If we haven't done it, we don't say we do it.

4. **Confident, not loud.** State capabilities plainly. "We build production-grade AI integrations" — not "We're revolutionizing AI." Confidence comes from specificity, not volume.

5. **Outcome-focused.** Frame everything in terms of what the client gets: a working pipeline, a deployed feature, a CI/CD setup that ships daily. Not what NorthOps is — what the client walks away with.

## Tone Guidelines

### Outreach (emails, LinkedIn messages)
- Conversational but professional. Short sentences. One clear ask.
- Lead with the client's problem, not NorthOps' capabilities.
- Tone: a senior engineer reaching out to a peer, not a salesperson pitching a product.
- Example: "How does your team deploy right now? We set up CI/CD pipelines in 3 days — happy to chat if that's useful."

### Website
- Clear and scannable. Short paragraphs, concrete examples, no walls of text.
- Lead each section with what the client gets, then explain how.
- Tone: a well-written README — informative, structured, no filler.
- Example: "Fixed-price MVPs starting at CAD $3,600. Scope defined, deliverables explicit, delivered in days."

### Proposals and SOWs
- Precise and structured. Numbered deliverables, explicit scope boundaries, clear acceptance criteria.
- No marketing language. The proposal is a technical document, not a sales deck.
- Tone: a scope document written by the engineer who will do the work.
- Example: "Deliverable 1: GitHub Actions CI pipeline (lint, test, build, deploy on merge to main). Estimated: 1.5 days."

### Social Media
- Short, factual, useful. Share what we build, how we build it, and what we learned.
- No engagement bait, no hot takes, no thought leadership posturing.
- Tone: an engineer posting about their work.
- Example: "Set up a RAG pipeline for a client this week. Embeddings + Pinecone + Claude. 5 days from kickoff to production."

| Use This | Because |
|----------|---------|
| "Senior engineer" | Accurate — both founders are senior |
| "Production-grade" | Distinguishes from prototypes and POCs |
| "Fixed-price MVP" | Concrete, low-commitment, fast to close |
| "Ship" / "deploy" / "deliver" | Action-oriented, outcome-focused |

| "Working software" | Outcome the client cares about |
| "Explicit scope" / "explicit deliverables" | Signals professionalism and reduces scope anxiety |

### Stack and Service References
- Use specific stack names: "Laravel," "React," "Drupal," "Go," "Rust" — not "modern frameworks"
- Use service line names exactly: "Web Application Engineering," "Content & Data Pipelines," "DevOps & CI/CD," "AI Engineering"
- Use "AI integration" or "AI Engineering" — not "AI solutions" or "AI-powered"

## Anti-Patterns

### Words and Phrases NorthOps Avoids

| Avoid | Why |
|-------|-----|
| "Solutions" / "synergy" / "leverage" | Corporate filler — says nothing |
| "Cutting-edge" / "best-in-class" / "world-class" | Unverifiable superlatives |
| "Revolutionize" / "transform" / "disrupt" | Saviour narrative — we build, we don't save |
| "Resources" (meaning people) | People are engineers, not resources |
| "Passionate" / "dedicated" / "committed" | Empty adjectives — show, don't tell |
| "End-to-end" | Vague — say what you actually cover |
| "Scalable" (without context) | Meaningless unless you say what scales and how |
| "Seamless" / "frictionless" | Nothing is seamless — be honest about trade-offs |
| "We're excited to..." | Filler — get to the point |
| "Trusted partner" / "strategic partner" | Earned, not claimed — let the work speak |
| "Rockstar" / "ninja" / "guru" | No. |

### Patterns to Avoid
- **Saviour narratives:** "We rescue struggling projects" — no. We do defined work on defined timelines.
- **Future promises:** "We're building toward..." — no. Talk about what exists now.
- **Vague value props:** "We help businesses grow" — no. Say what you build and what outcome it produces.
- **Passive voice in deliverables:** "The pipeline will be configured" → "We configure the pipeline."
- **Hedging:** "We can potentially help with..." → "We build X. Here's the scope."

## Messaging Pillars

### 1. Senior-Only Engineering
Every hour is a senior engineer writing production code. No juniors, no PMs, no coordination tax. The people who scope the work are the people who build it.

### 2. Speed and Clarity
Fixed-price MVPs delivered in 3–10 days. Explicit scope, explicit deliverables, explicit acceptance criteria. Clients know what they're getting, what it costs, and when it arrives.

### 3. Full-Stack, One Team
Backend, frontend, infrastructure, CI/CD, and AI integration — from one two-person team. No subcontracting, no vendor coordination, no handoff gaps.

### 4. Low-Friction Start
Simple SOW, 50% deposit via e-transfer, work begins. The first paid engagement can close in days. No procurement portals, no six-week onboarding, no vendor questionnaires.

### 5. AI as Engineering, Not Experiment
Production-grade AI integration — RAG, embeddings, agent workflows — delivered as working features, not research projects. Built by a senior engineer, tested, and deployed.
